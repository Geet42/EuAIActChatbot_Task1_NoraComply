## Part 1 — What a "ChatGPT Wrapper" Actually Is (and Why This Isn't One)

A ChatGPT wrapper does exactly this:

```
User question → send to OpenAI API → return answer
```

The LLM answers from its **training weights** — a static snapshot of the internet frozen at a cutoff date. For general questions this is fine. For compliance questions this is dangerous, for three reasons:

| Problem | Why it matters for compliance |
|---|---|
| **Hallucination** | GPT-4o will confidently state "Article 26(3) requires X" even if Article 26(3) says something completely different |
| **Stale law** | The EU AI Act was finalised in 2024. LLM training data is patchy on post-2023 legislation |
| **No auditability** | If a regulator asks "why did your system give this advice?" a wrapper has no answer |

This system does not ask ChatGPT what the law says. It **tells** ChatGPT exactly what the law says, then asks it to reason over that text. The LLM is used only for its language capability — structuring, explaining, and connecting clauses. The knowledge comes from the official regulation, loaded by the retrieval pipeline.

## Part 2 — How RAG Works: Every Step Mapped to the Actual Code

RAG has three components: **a knowledge base**, **a retrieval engine**, and **a grounded generator**. Here is exactly how each is implemented.

---

### Step 1 — The Knowledge Base (`lib/knowledge-base.ts`)

Before any user query arrives, a decision is made: what does the system know, and how is it structured?

Every fact in this system comes from one of two official sources:
- **EUR-Lex CELEX:32024R1689** — the Official Journal publication of the EU AI Act
- **artificialintelligenceact.eu** — the per-article mirror of that text

Each piece of knowledge is stored as a **chunk** — a focused, self-contained unit covering exactly one concept:

```typescript
{
  id: "art26-6",
  source: "Article 26, Paragraph 6",
  title: "Deployer Obligation: Log Retention — Minimum 6 Months",
  sourceUrl: "https://artificialintelligenceact.eu/article/26/",
  content: `"Deployers of high-risk AI systems shall keep the logs automatically 
  generated by that high-risk AI system to the extent such logs are under their 
  control, for a period appropriate to the intended purpose of the high-risk AI 
  system, of at least six months, unless provided otherwise in applicable Union 
  or national law..." (Article 26(6))`
}
```

**Why this matters for RAG accuracy:**  
Chunk design is the most consequential decision in a RAG system. Too large and retrieval becomes noisy — a chunk about Article 26(1) and Article 26(6) will be retrieved for both questions but only partially relevant to each. Too small and context is lost. Each chunk here covers one paragraph or one tightly related group of obligations. This is why the knowledge base has 27 separate chunks across 6 articles rather than 6 large article summaries.

The knowledge base currently covers:
- Article 5 (prohibited practices)
- Article 6 (high-risk classification rules)
- Annex III Point 4(a) and 4(b) (employment AI — verbatim)
- Article 9 (risk management lifecycle)
- Article 13 (transparency obligations on providers)
- Article 14 (human oversight — all paragraphs)
- Article 26 (all 12 paragraphs of deployer obligations)
- Article 27 (all 5 paragraphs of FRIA)
- Article 113 (enforcement timeline — exact dates)
- GPAI obligations (Articles 51–55)
- Ireland-specific: WRC as market surveillance authority

### Step 2 — The Embedding Step (Query → Vector)

When a user submits a question, the system calls OpenAI's `text-embedding-3-small` model to convert the question into a **1536-dimensional vector**:

```typescript
// app/api/chat/route.ts — lines 51–55
const queryEmbeddingResponse = await openaiClient.embeddings.create({
  model: "text-embedding-3-small",
  input: latestMessage,
});
const queryEmbedding = queryEmbeddingResponse.data[0].embedding;
```

This vector is not a bag of words. It is a geometric point in a 1536-dimensional semantic space where meaning, not keywords, determines proximity. The sentence "how long must I keep logs?" and "what is the log retention requirement?" will produce vectors that are very close together, even though they share no keywords.

This is why the retrieval does not fail on synonyms, paraphrasing, or different phrasings of the same legal question.

---

### Step 3 — The Retrieval Step (Query Vector → Relevant Chunks)

Every chunk in the knowledge base is also embedded into the same vector space. This happens once on cold start and is cached in memory:

```typescript
// app/api/chat/route.ts — lines 26–42
async function getEmbeddedChunks(openaiClient: OpenAI) {
  if (embeddedChunksCache) return embeddedChunksCache;

  const texts = KNOWLEDGE_BASE.map((chunk) => `${chunk.title}. ${chunk.content}`);
  const response = await openaiClient.embeddings.create({
    model: "text-embedding-3-small",
    input: texts,
  });

  embeddedChunksCache = KNOWLEDGE_BASE.map((chunk, index) => ({
    ...chunk,
    embedding: response.data[index].embedding,
  }));
  return embeddedChunksCache;
}
```

Each chunk is now a point in the same 1536-dimensional space as the user's query.

Retrieval works by computing **cosine similarity** between the query vector and every chunk vector:

```typescript
// lib/rag-utils.ts — lines 9–16
export function cosineSimilarity(a: number[], b: number[]): number {
  const dot = a.reduce((sum, val, i) => sum + val * b[i], 0);
  const magnitudeA = Math.sqrt(a.reduce((sum, val) => sum + val * val, 0));
  const magnitudeB = Math.sqrt(b.reduce((sum, val) => sum + val * val, 0));
  if (magnitudeA === 0 || magnitudeB === 0) return 0;
  return dot / (magnitudeA * magnitudeB);
}
```

Cosine similarity measures the **angle** between two vectors. A score of 1.0 means the vectors point in the same semantic direction (identical meaning). A score of 0 means they are orthogonal (unrelated). The system sorts all 27 chunks by this score and takes the top 6:

```typescript
// lib/rag-utils.ts — lines 27–40
export function rankChunksBySimilarity(
  queryEmbedding: number[],
  chunkEmbeddings: Array<{...}>,
  topK: number = 6
): ScoredChunk[] {
  const scored = chunkEmbeddings.map((chunk) => ({
    ...chunk,
    score: cosineSimilarity(queryEmbedding, chunk.embedding),
  }));
  return scored.sort((a, b) => b.score - a.score).slice(0, topK);
}
```

**Why topK = 6:** Each chunk is one paragraph. A realistic compliance question may span 2–3 related articles. 6 chunks is enough to cover Article 26 + Article 27 + Annex III simultaneously without overwhelming the context window with noise.

---

### Step 4 — The Grounded System Prompt

The top 6 chunks are injected into the LLM's system prompt **before** any user message is processed:

```typescript
// lib/rag-utils.ts — buildSystemPrompt()
const context = relevantChunks
  .map((chunk) => 
    `[${chunk.source}]\nTitle: ${chunk.title}\nOfficial source: ${chunk.sourceUrl}\n\n${chunk.content}`)
  .join("\n\n---\n\n");

return `You are a compliance assistant...
Your answers are grounded exclusively in the following retrieved sections 
of the EU AI Act. Do not speculate beyond this context...

RETRIEVED CONTEXT (from official EU AI Act text):
${context}
...`;
```

The LLM receives the verbatim legal text as context. It is explicitly instructed:
- Do not speculate beyond this context
- Cite the specific Article and paragraph number
- Use quotation marks when quoting verbatim text
- List source URLs at the end of every answer

The LLM's role is now **reasoning and explanation**, not knowledge recall. It has to work with what the retrieval layer gave it. This is the architectural constraint that separates this system from a wrapper.

---

### Step 5 — The Streamed Response

The response is streamed token-by-token back to the browser using the Vercel AI SDK:

```typescript
// app/api/chat/route.ts — lines 65–71
const result = streamText({
  model: openai("gpt-4o"),   // gpt-4o, not gpt-4o-mini — legal reasoning quality matters
  system: systemPrompt,
  messages,
});
return result.toDataStreamResponse();
```

`gpt-4o` was specifically chosen over `gpt-4o-mini` because legal reasoning requires the model to hold multiple clause relationships simultaneously. The 3× cost difference is justified for a compliance product where a wrong answer has legal consequences.

## Part 3 — Why RAG Specifically Matters for a Compliance Product

Most applications can tolerate occasional hallucination. A compliance product cannot. Here is why RAG is not just a nice-to-have — it is the only architecturally sound approach for this domain.

### 3.1 — The Law Does Not Change on the LLM's Schedule

The EU AI Act's high-risk deployer obligations (Article 26, Article 27) apply from **2 August 2026**. This date appears in Article 113 of the regulation. An LLM trained before the Act was finalised may have conflicting information from drafts, commentary, and political coverage. A RAG system reads from the final published regulation. When the Commission publishes an amendment, you update one chunk. The LLM does not need to be retrained.

### 3.2 — Hallucinated Legal Advice Creates Real Liability

Consider the difference:

**ChatGPT (wrapper) asked:** "Do I need to do a FRIA as a private recruitment agency?"

Likely answer: *"Yes, under Article 27 of the EU AI Act, all deployers of high-risk AI systems must conduct a Fundamental Rights Impact Assessment before deployment."*

This is **wrong**. Article 27(1) mandates the FRIA for public bodies and private entities providing public services. A private recruitment agency is not automatically within scope. A company acting on the wrapper's advice would spend resources on a document they may not legally be required to file — or worse, would treat it as a box-ticked substitute for other obligations they do have.

**This system retrieves the exact text of Article 27(1) verbatim** and the LLM is constrained to reason from it:

> *"Prior to deploying a high-risk AI system referred to in Article 6(2)... deployers that are bodies governed by public law, or are private entities providing public services, and deployers of high-risk AI systems referred to in points 5(b) and (c) of Annex III, shall perform an assessment..."*

The system then correctly explains: private recruitment agencies are not in automatic scope, but should conduct one for audit readiness and good-practice compliance. That is a materially different — and legally accurate — answer.

### 3.3 — Auditability: Every Answer Has a Chain of Evidence

When a compliance team is audited, they need to demonstrate that their process was based on correct legal understanding. With a RAG system, every answer has a complete chain of evidence:

1. User asked: *"What is the log retention requirement?"*
2. Query embedding matched: `art26-6` with cosine similarity 0.91
3. Injected context: verbatim text of Article 26(6)
4. LLM output: answer citing Article 26(6), minimum 6 months
5. Source URL: https://artificialintelligenceact.eu/article/26/

A wrapper produces step 4 only. You cannot show a regulator what the system was looking at when it gave its advice.

### 3.4 — Scope Control

A compliance product should not answer off-topic questions. Because the knowledge base is curated, the system's knowledge is bounded. Ask it about GDPR Article 5 (outside the knowledge base) — it says so and directs the user to the official text. A wrapper answers everything, confidently, with no signal about where its confidence ends.

## Part 5 — What Would Make This Even Better in Production

Understanding the limits of the current system is itself evidence of deep RAG knowledge.

| Current approach | Production upgrade | Why |
|---|---|---|
| In-memory embedding cache | pgvector on Supabase — pre-computed embeddings stored in a vector DB | Cold start time eliminated; scales to thousands of chunks |
| Manual knowledge base | EUR-Lex SPARQL endpoint + monthly cron diff | Regulation changes are detected automatically; human review gate before any update ships |
| Cosine similarity only (bi-encoder) | Add a cross-encoder re-ranking step after initial retrieval | Initial retrieval optimises for speed; cross-encoder re-ranks top-20 for precision before top-6 are injected |
| Single-turn RAG | Conversation-aware retrieval — embed the last 3 turns together | Follow-up questions ("what about Article 27 then?") retrieve correctly without repeating context |
| No confidence signal | Return cosine similarity scores in the UI | Users see when the system is highly confident (score > 0.85) vs borderline (score < 0.6) |
| Static chunk structure | Hierarchical chunking: parent = full article, child = paragraph | Retrieve at paragraph level, inject parent context if score is ambiguous |
