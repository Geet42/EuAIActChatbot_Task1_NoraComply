## Part 1 — What a wrapper actually is

A ChatGPT wrapper does this:

```
user question → OpenAI API → answer
```

That's it. The model answers from its training weights — a static snapshot of the internet with a cutoff date. For most questions that's fine. For compliance questions it's genuinely dangerous:

| Problem | Why it matters |
|---|---|
| **Hallucination** | GPT-4o will confidently cite "Article 26(3) requires X" when Article 26(3) says something completely different |
| **Stale law** | The EU AI Act was finalised in 2024. LLM training data on post-2023 legislation is patchy and often based on drafts, not the final text |
| **No audit trail** | If a regulator asks "where did your system get that advice from?", a wrapper has no answer |

What I built doesn't ask ChatGPT what the law says. I tell it exactly what the law says — by retrieving the relevant verbatim text first — and then ask it to reason over that. The LLM handles language. The knowledge comes from the regulation.

## Part 2 — How RAG works, mapped to the actual code

RAG has three parts: a knowledge base, a retrieval engine, and a grounded generator. Here's each one.

---

### Step 1 — Knowledge base (`lib/knowledge-base.ts`)

Before any user message arrives, I had to decide: what does the system know, and how is it chunked?

Everything comes from two official sources:
- **EUR-Lex CELEX:32024R1689** — the Official Journal publication
- **artificialintelligenceact.eu** — per-article mirror of the same text

I split the regulation into 27 chunks. Each chunk covers exactly one paragraph or one tightly related concept:

```typescript
{
  id: "art26-6",
  source: "Article 26, Paragraph 6",
  title: "Deployer Obligation: Log Retention — Minimum 6 Months",
  sourceUrl: "https://artificialintelligenceact.eu/article/26/",
  content: `"Deployers of high-risk AI systems shall keep the logs automatically 
  generated by that high-risk AI system... for a period appropriate to the 
  intended purpose of the high-risk AI system, of at least six months..." (Article 26(6))`
}
```

Chunk design is probably the most important decision in any RAG system. If a chunk covers half an article, it gets retrieved for multiple unrelated questions and adds noise. One paragraph per chunk means retrieval is precise — the chunk about log retention only comes up when you ask about log retention.

Coverage: Article 5, 6, 9, 13, 14, 26 (all 12 paragraphs), 27 (all 5 paragraphs), 113, Annex III Point 4(a)/(b), GPAI obligations, Ireland WRC.

### Step 2 — Embedding the query

When a user sends a question, I embed it using `text-embedding-3-small`:

```typescript
// app/api/chat/route.ts
const queryEmbeddingResponse = await openaiClient.embeddings.create({
  model: "text-embedding-3-small",
  input: latestMessage,
});
const queryEmbedding = queryEmbeddingResponse.data[0].embedding;
```

This converts the question into a 1536-dimensional vector. The important thing here is that this isn't keyword matching — it's semantic. "How long must I keep logs?" and "what's the retention period for system records?" produce vectors that sit close together in that space, even though they share no keywords. That's why the retrieval doesn't break on synonyms or rephrasing.

---

### Step 3 — Retrieval by cosine similarity

All 27 chunks are also embedded (once on cold start, cached in memory). Retrieval is just computing the angle between the query vector and each chunk vector:

```typescript
// lib/rag-utils.ts
export function cosineSimilarity(a: number[], b: number[]): number {
  const dot = a.reduce((sum, val, i) => sum + val * b[i], 0);
  const magnitudeA = Math.sqrt(a.reduce((sum, val) => sum + val * val, 0));
  const magnitudeB = Math.sqrt(b.reduce((sum, val) => sum + val * val, 0));
  if (magnitudeA === 0 || magnitudeB === 0) return 0;
  return dot / (magnitudeA * magnitudeB);
}
```

A score of 1.0 means the vectors point in the same direction — identical meaning. 0 means they're unrelated. I sort all 27 chunks by this score and take the top 6:

```typescript
export function rankChunksBySimilarity(
  queryEmbedding: number[],
  chunkEmbeddings: Array<{...}>,
  topK: number = 6
): ScoredChunk[] {
  const scored = chunkEmbeddings.map((chunk) => ({
    ...chunk,
    score: cosineSimilarity(queryEmbedding, chunk.embedding),
  }));
  return scored.sort((a, b) => b.score - a.score).slice(0, topK);
}
```

I use topK = 6 because a realistic compliance question often spans 2–3 related articles. 6 chunks is enough to cover Article 26 + Article 27 + Annex III simultaneously without flooding the context window.

---

### Step 4 — Grounded system prompt

The top 6 chunks go into the system prompt before the LLM sees the user's message:

```typescript
// lib/rag-utils.ts — buildSystemPrompt()
const context = relevantChunks
  .map((chunk) => 
    `[${chunk.source}]\nTitle: ${chunk.title}\nOfficial source: ${chunk.sourceUrl}\n\n${chunk.content}`)
  .join("\n\n---\n\n");

return `You are a compliance assistant...
Your answers are grounded exclusively in the following retrieved sections 
of the EU AI Act. Do not speculate beyond this context...

RETRIEVED CONTEXT (from official EU AI Act text):
${context}`;
```

The LLM is explicitly told: don't speculate, cite the exact Article and paragraph, quote verbatim text in quotation marks, list source URLs at the end. The LLM's job is reasoning and explanation — not remembering law it might have seen in training.

---

### Step 5 — Streaming the response

```typescript
// app/api/chat/route.ts
const result = streamText({
  model: openai("gpt-4o"),
  system: systemPrompt,
  messages,
});
return result.toDataStreamResponse();
```

I use gpt-4o rather than gpt-4o-mini. Legal reasoning requires holding multiple clause relationships in context at once — the smaller model cuts corners. For a compliance product where a wrong answer has real consequences, the cost difference is worth it.

## Part 3 — Why RAG specifically matters here

### The law doesn't change on the LLM's schedule

Article 26 and Article 27 obligations apply from 2 August 2026. This date is in Article 113 of the regulation. An LLM trained on internet data may have picked up conflicting information from draft versions, political commentary, or summaries. I pull from the final published text. When something changes, I update one chunk — no retraining required.

### Hallucinated legal advice creates real liability

Here's a concrete example. Ask a generic LLM: "Does a private recruitment agency have to do a FRIA?"

A common answer: *"Yes, under Article 27, all deployers of high-risk AI must conduct a Fundamental Rights Impact Assessment before deployment."*

This is wrong. Article 27(1) mandates the FRIA for **public bodies** and **private entities providing public services**. A standard private recruitment agency is not automatically in scope.

My system retrieves the exact text of Article 27(1) and the LLM has to reason from that. The correct answer — with the nuance about private sector scope — comes out because the source text was actually present in context.

### Every answer has a chain of evidence

When a compliance team is audited, they need to show their process was based on correct legal understanding. This system produces a complete trace:

1. User asked: *"What is the log retention requirement?"*
2. Query embedding matched `art26-6` — cosine similarity ~0.91
3. Injected context: verbatim Article 26(6) text
4. LLM output: answer citing Article 26(6), minimum 6 months
5. Source link: https://artificialintelligenceact.eu/article/26/

A wrapper produces step 4 only. You can't show a regulator what the system was looking at.

### Scope control

Because the knowledge base is curated, the system's knowledge is bounded. Ask it something outside its coverage and it says so — it doesn't confidently answer from training data. That's a feature, not a limitation.

## Part 5 — What I'd build next in production

Knowing the limits is as important as knowing the strengths.

| Current | Production version | Reason |
|---|---|---|
| In-memory embedding cache | pgvector on Supabase | Eliminates cold-start time; scales to thousands of chunks |
| Manual knowledge base | EUR-Lex SPARQL + monthly diff | Amendment detection without manual monitoring |
| Cosine similarity only | Cross-encoder re-ranking after initial retrieval | Better precision on borderline cases |
| Single-turn RAG | Multi-turn retrieval — embed last 3 turns together | Follow-up questions retrieve correctly without losing context |
| No confidence signal | Return cosine scores in the UI | Users see when the system is highly confident vs borderline |
| Flat chunk structure | Hierarchical chunking (article + paragraph) | Retrieve at paragraph level, fall back to article context if score is ambiguous |
